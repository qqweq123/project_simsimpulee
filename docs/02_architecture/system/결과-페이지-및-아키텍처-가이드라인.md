# Mellow Wave 결과 페이지 및 아키텍처 가이드라인 (v2.0)

> **작성자**: Mellow Wave 총괄 수석 엔지니어 (@general-engineer)
> **목적**: 기획 의도와 프로그래밍 공학적 원칙을 통합하여, 모든 에이전트와 기획자가 일관성 있게 참조할 수 있는 마스터 가이드라인

본 문서는 초기 기획안이었던 `결과_페이지_구성에_관해.txt`의 핵심 요구사항과 최근 고도화 작업을 통해 정립된 기획, 디자인, 데이터 아키텍처 원칙을 하나로 통합한 것입니다. 클린 코드를 지향하고 뛰어난 유지보수성을 확보하기 위해 아래 원칙을 최우선으로 준수해야 합니다.

---

## 1. 결과 페이지 기본 구성 및 레이아웃 (Layout Architecture)

결과 페이지는 심미성이 매우 뛰어나야 하며, 각 테스트별 개성 있는 모양새와 컨셉을 담아내야 합니다. 기본적으로 다음 크게 3가지 섹션으로 구성됩니다.

1. **결과 이미지 및 유형명 섹션**
   - 테스트 결과에 따른 텍스트 출력 및 메인 아트워크가 표시됩니다.
2. **결과 설명 섹션**
   - 200~400자 이내의 한글 설명이 표시됩니다. (차후 인도네시아어, 영어, 일본어, 중국어 등 다국어 지원 예정)
3. **공유 및 내비게이션 섹션**
   - 소셜 미디어 공유 버튼과 다른 테스트로 이동하는 버튼, 다시 테스트하기 버튼이 렌더링 됩니다.
   다만 다시테스트 하기 버튼은 해당 심리테스트 콘셉트에 맞는 표현으로 나타내야 합니다. (예 무인도 테스트에서는 '다시 표류하기🏝️' 등)
4. **광고 섹션**
   - 스폰서 광고가 표시됩니다.
---

## 2. 프론트엔드 (UI/UX) 핵심 원칙

### 2-1. 트렌디한 디자인
- **일관된 톤앤매너**: 질문 페이지부터 결과 페이지까지 배경색과 질문지 및 결과지의 톤이 **매끄럽게 이어져야(Seamless)** 합니다.
- **독창적인 디자인**: 각 테스트는 테스트의 콘셉트와 맞게 독창적인 디자인을 가져야 합니다.

### 2-2. 타겟팅 최우선 뷰포트 (Above the Fold) 설계
- **한 줄 철학**: 결과 제목(`.result-title`)은 모바일/데스크톱 불문하고 **최대한 한 줄 내에 들어오도록** `whitespace-nowrap`, `tracking-tighter` 등으로 코딩합니다.
- **광고 배치 최적화**: 
  - 핵심 스폰서 영역은 스크롤 없이 보이는(Above the fold) 곳으로 전진 배치합니다.
  - 서브/인라인 광고는 공유 버튼 바로 밑으로 빼서 UI 동선을 해치지 않게 합니다.

### 2-3. 로딩과 전환 (Interstitial Ad Flow)
- 테스트 완료 후 100vh 전면을 덮는 **전면 광고 및 로딩 스크린(`lock-overlay`)**을 삽입합니다. 최소 4초간 머무르게 하여 체류 시간과 구조적 광고 효율을 극대화한 후 패스해야 합니다.

## 3. 데이터 아키텍처 및 퍼포먼스 (Data & Performance)

### 3-1. 의존성 격리 및 SVG 최적화 (공유 로직 포함)
- 카카오, 페북, X, 인스타, 쓰레드 공유 버튼은 무거운 픽셀 이미지나 외부 라이브러리 대신, **순수 웹표준 Inline SVG 벡터 로고**로 삽입합니다.
- 인스타그램 및 페이스북 공유 시 기본적으로 결과 페이지 **이미지가 썸네일로 첨부**되어야 하며, 인스타그램은 트렌디한 이미지 크기에 맞춰야 합니다.

### 3-2. CSS-Only Background (데이터 최적화)
- 결과 페이지 전체 분위기를 좌우하는 백그라운드는 무거운 거대 이미지가 아닌 CSS 파일(`themes/[테마명].css`)에 정의된 네이티브 브라우저 그래픽(`linear-gradient` 등)을 우선 채택합니다.
- 단, 생성된 캐릭터 이미지는 임시 저장(Storage) 환경을 거쳐, 런칭 단계에서는 **반드시 Supabase 등 정식 클라우드 스토리지에 이관 후 서빙**해야 합니다. (기본 480x480 규격)

---

## 4. 아트워크 에셋 생성 규칙 (Image Generation)

### 4-1. 절대적 일관성과 다이나믹 렌더링
- **Base Reference 기법**: 여러 개의 결과 캐릭터 이미지를 뽑을 때 최초 1개의 기준 이미지를 100% 스타일 피드로 잡아 전부 똑같은 그림체와 비율로 생성되게 합니다.
- **Dynamic Action & Context**: 무표정 생성을 배제하고 결과값에 맞는 환한 미소, 정확한 의상, 행동 포즈, 그리고 해당 테스트(예: 무인도) 컨셉에 완벽하게 일치하는 풍경 배경을 꽉 차게 렌더링해야 합니다. 다만, 배경의 동일은 지양하며, 반드시 *같은 식생의 배경으로 캐릭터의 콘셉에 알맞은 다른 배경을 렌더링 하는 것을 목표*로 합니다.
- **캐릭터 일관성 유지** 결과 캐릭터 이미지는 캐릭터 일관성을 반드시 지키며 주변 식생 일관성도 지켜야 합니다.
---

## 5. 공유 및 링크 네비게이션 프로토콜

### 5-1. 동적 공유 API 연동
- 모든 공유 버튼(`.shareSNS`)은 단순 링크 열기가 아닌, 실제 공유 동작을 타겟으로 합니다.
- **모바일**: `navigator.share` (Web Share API)를 최우선 호출하여 네이티브 지원 팝업 연동.
- **PC/Fallback**: URL Intent 쿼리를 통한 데스크톱 소셜 공유 팝업 호출 등.

### 5-2. 다른 테스트 하러 가기 (Navigation)
- **랜덤/전체 라우팅**: '다른 테스트 하러 가기'는 **반드시 지정된 '전체 테스트 페이지(`/src/pages/tests/test_list.html`)' 컴포넌트로 정확히 리다이렉트** 되어야 합니다.
