1. 아키텍처 개요 (Architecture Overview)
---------------------------------------------------------------------------
본 프로젝트는 **Vite** 빌드 시스템과 **Feature-Sliced Design (FSD)** 아키텍처 원칙을
경량화하여 적용하고 있습니다.

  [Root] src/
  ├── core/        : 전역 공유 로직 (ThemeManager, API, Store)
  ├── features/    : 비즈니스 로직 및 데이터 (독립적 기능 단위)
  ├── pages/       : 라우팅 및 뷰 계층 (HTML 진입점)
  └── shared/      : 재사용 가능한 UI 컴포넌트 및 유틸리티

2. 데이터 관리 원칙 (Data Management Principles)
---------------------------------------------------------------------------
**"Code as Single Source of Truth" (SSOT)**

- **원칙**: 구현이 완료된 테스트의 콘텐츠와 로직은 오직 **코드(`data.js`)**만이 진실입니다.
- **Archive**: 초기 기획 텍스트 파일은 `docs/archive/`에 보관하되, 유지보수하지 않습니다.
- **Design**: 복잡한 로직(점수표 등)은 `docs/design/`에 별도 문서화하여 개발 가이드로 삼습니다.

3. Import 경로 전략 (Import Path Strategy)
---------------------------------------------------------------------------
유지보수성과 가독성을 위해 아래 규칙을 엄격히 준수합니다.

(1) **@/ Alias 사용 의무화**: 다른 계층(features, core) 참조 시 필수
    - ✅ `import { ... } from '@/features/tests/demon/data.js'`
    - ❌ `import { ... } from '../../../features/tests/demon/data.js'`

(2) **상대경로 제한적 허용**: **동일 디렉토리 내** 참조 시에만 허용
    - ✅ `import { ... } from './data.js'` (같은 폴더)

4. 테스트 모듈 표준 구조 (Standard Test Module Structure)
---------------------------------------------------------------------------
모든 심리 테스트는 `features/tests/{test_name}/` 디렉토리 내에
다음 3가지 파일을 의무적으로 포함해야 합니다.

  (1) `data.js`
      - 질문 배열 (`questions`), 결과 객체 (`results`) export
      - 순수 데이터만 포함 (로직 최소화)

  (2) `question.js`
      - `init{TestName}Test()` 함수 export
      - 질문 렌더링, 진행 바 업데이트, 답변 선택 로직 담당

  (3) `result.js`
      - `init{TestName}Result()` 함수 export
      - `ThemeManager`를 사용하여 테마 로딩 및 결과 데이터 바인딩

5. 문서 분류 체계 (Documentation Structure)
---------------------------------------------------------------------------
- `docs/planning/`: 프로젝트 로드맵, 기획서, 기술 의사결정 기록
- `docs/design/`: 시스템 설계도, 매핑 테이블, 알고리즘 명세 (Active)
- `docs/archive/`: 구현 완료된 구 기획서, 원본 텍스트 데이터 (Inactive)
